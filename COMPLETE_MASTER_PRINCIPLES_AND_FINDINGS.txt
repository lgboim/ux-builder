A. NIELSEN NORMAN GROUP (NN/g) 10 USABILITY HEURISTICS
These are the core qualitative framework for expert evaluation of interfaces.

1. Visibility of System Status
   - The system should always keep users informed about what is going on through 
     appropriate feedback within a reasonable time.
   - Examples: progress bars, loading spinners, cart badges showing item count
   - Binary Application: Is there a progress indicator? Yes/No

2. Match Between System and Real World
   - The system should speak the users' language, with words, phrases, and 
     concepts familiar to the user, rather than system-oriented terms or jargon.
   - Example: "Security Code" instead of "CVV/CID"
   - Example: "Apt. #, Suite, Floor" instead of "Address Line 2"
   - Binary Application: Are familiar user terms used? Yes/No

3. User Control and Freedom
   - Users often choose system functions by mistake and need a clearly marked 
     "emergency exit" to leave the unwanted state without an extended dialogue.
   - Examples: Undo/Redo, Cancel buttons, ability to go back
   - Example: Can users "Undo" a deletion?
   - Binary Application: Is there a marked emergency exit? Yes/No

4. Consistency and Standards
   - Users should not have to wonder whether different words, situations, or 
     actions mean the same thing.
   - Follow platform conventions and internal site standards.
   - Jakob's Law: Users prefer your site to work the same way as all the other 
     sites they already know.
   - Example: Cart icon in the top right
   - Example: Search field looks like a standard input
   - Binary Application: Are platform conventions followed? Yes/No

5. Error Prevention
   - Even better than good error messages is a careful design which prevents a 
     problem from occurring in the first place.
   - Design should prioritize preventing problems before they occur.
   - Examples: constraints, sensible defaults, confirmation before risky actions
   - Example: Block invalid characters in numeric-only fields
   - Example: Never use "Reset" buttons (risk of catastrophic data loss)
   - Binary Application: Are error-prone conditions eliminated? Yes/No

6. Recognition Rather Than Recall
   - Minimize the user's memory load by making objects, actions, and options 
     visible.
   - The user should not have to remember information from one part of the 
     dialogue to another.
   - Examples: recently viewed items shown, autocomplete suggestions
   - Binary Application: Are all relevant options visible? Yes/No

7. Flexibility and Efficiency of Use
   - Accelerators—unseen by the novice user—may often speed up the interaction 
     for the expert user.
   - Examples: keyboard shortcuts, gestures, "Select All" functionality
   - Binary Application: Are accelerators provided? Yes/No

8. Aesthetic and Minimalist Design
   - Dialogues should not contain information which is irrelevant or rarely needed.
   - Every extra unit of information competes with the relevant units of information.
   - Example: Checkout pages should be free of promotional banners and unrelated links
   - Binary Application: Is irrelevant information removed? Yes/No

9. Help Users Recognize, Diagnose, and Recover from Errors
   - Error messages should be expressed in plain language (no error codes).
   - Precisely indicate the problem.
   - Constructively suggest a solution.
   - Example: "Invalid Email" instead of "Error 0x442"
   - Example: "Include an @ symbol" as a constructive suggestion
   - Binary Application: Are errors in plain language with solutions? Yes/No

10. Help and Documentation
    - Even though it is better if the system can be used without documentation, 
      it may be necessary to provide help and documentation.
    - Help should be easy to search.
    - Help should be focused on tasks.
    - Help should not be overly large.
    - Example: Searchable FAQ, contextual help links during complex steps
    - Binary Application: Is documentation easy to search and task-focused? Yes/No

B. INTERACTION AND BIOMECHANICAL LAWS

1. Fitts's Law
   Formula: T = a + b log₂(1 + D/W)
   Where:
   - T = Time to acquire target
   - D = Distance to target
   - W = Width/Size of target
   - a, b = Constants based on device and user
   
   Key Principles:
   - Target Size (W): Larger targets are easier and faster to hit
   - Distance (D): Shorter distances to targets result in faster acquisition
   - Prime Pixel: The fastest pixel to click is the one the cursor is already on
   - Magic Pixels: The four corners and edges of a screen act as "infinite" targets 
     because the cursor cannot overshoot them
   - Example: Top-right corner for close/exit button is optimal for acquisition speed
   - Tactile Precision: Human finger pads are approximately 9mm wide
   - Binary Application: Are targets appropriately sized for acquisition? Yes/No

2. Miller's Law
   - The average person can only keep 7 (plus or minus 2) items in working memory
   - Therefore: 7 ± 2 items = 5–9 items maximum
   - Violation: Unbroken 16-digit credit card strings (e.g., "4532123456789012")
   - Solution: Auto-format credit cards to break them visually (4532-1234-5678-9012)
   - Binary Application: Is information grouped into chunks of 5–9? Yes/No

3. Jakob's Law
   - Users spend most of their time on other sites.
   - They prefer your site to work the same way as all the other sites they 
     already know.
   - Implication: Follow industry standards and conventions.
   - Binary Application: Does the interface follow established patterns? Yes/No

4. Hick's Law
   - The time it takes to make a decision increases with the number and 
     complexity of choices.
   - Decision time increases logarithmically with choice complexity.
   - Implication: Reduce the number of visible options.
   - Example: Use progressive disclosure (hide advanced options behind links)
   - Binary Application: Is choice complexity minimized? Yes/No

5. BJ Fogg's Behavior Model
   - Behavior occurs when three factors align: Motivation + Ability + Trigger
   - Implication for forms: Reducing form fields increases the "Ability" component
   - Example: More fields = lower ability = lower completion rates
   - Binary Application: Are the three components optimized? Yes/No

6. Endowed Progress Effect
   - Users are more likely to complete a task if they feel progress has already 
     been made.
   - Example: A progress bar starting at "Step 1: Done" upon landing increases 
     completion rates
   - Example: Pre-filling form fields shows progress has been made
   - Binary Application: Is initial progress demonstrated? Yes/No

7. Commitment and Consistency Principle
   - Users who answer easy questions at the start of a form are more likely to 
     finish invasive questions (like salary or credit card) at the end.
   - Implication: Start with low-friction questions.
   - Example: Name and email (low friction) before billing address (higher friction)
   - Binary Application: Are easy questions asked first? Yes/No

8. Saccadic Scanning
   - Eye-tracking research shows users scan multicolumn forms in a Z-pattern
   - This Z-pattern breaks vertical momentum and reduces scanning efficiency
   - Finding: Single-column forms are 15.4 seconds faster than multi-column ones
   - Implication: Use single-column layouts for forms
   - Binary Application: Is a single-column layout used? Yes/No

C. ISO 9241-110 DIALOGUE PRINCIPLES

1. Suitability for the Task
   - The dialogue must support the user in completion of their task
   - Binary Application: Does the interface enable task completion? Yes/No

2. Self-Descriptiveness
   - The dialogue is immediately comprehensible through feedback and explanations
   - Users should not have to guess what to do next
   - Binary Application: Is functionality immediately clear? Yes/No

3. Conformity with User Expectations
   - The dialogue is consistent with the user's predictable behavior patterns
   - Binary Application: Does it match user mental models? Yes/No

4. Suitability for Learning
   - The user is guided through learning phases
   - Progressive disclosure and contextual help support learning
   - Binary Application: Is learning path clear? Yes/No

5. Controllability
   - The user can maintain control over the speed and sequence of the interaction
   - Users should be able to slow down, skip steps, or navigate non-linearly
   - Binary Application: Does user control speed/sequence? Yes/No

6. Error Tolerance
   - The intended result is achieved with minimal corrective action despite errors
   - Example: Auto-correcting typos, allowing undo, providing suggestions
   - Binary Application: Can errors be recovered easily? Yes/No

7. Suitability for Individualization (2006) / User Engagement (2020)
   - The interface can be adjusted to the user's needs and preferences
   - Example: Font size adjustment, theme selection, language preference
   - Binary Application: Can users customize the interface? Yes/No

E. W3C WEB CONTENT ACCESSIBILITY GUIDELINES (WCAG 2.2) - SPECIFIC CRITERIA

1. Contrast Ratios (SC 1.4.3)
   - Body Text: 4.5:1 ratio (Level AA), 7:1 (Level AAA)
   - Large Text (18pt or larger, or 14pt bold or larger): 3:1 (Level AA), 4.5:1 (Level AAA)
   - Non-Text UI (buttons, borders, icons): 3:1 ratio
   - Note: Ratios cannot be rounded. 4.499:1 is a failure.
   - Binary Pass: Ratio ≥ 4.5:1 for body text? Yes/No

2. Non-Text Contrast (SC 1.4.11)
   - UI components and graphical objects must have a 3:1 ratio against background colors
   - Binary Pass: UI elements have 3:1 contrast? Yes/No

3. Touch Target Size (SC 2.5.5 / 2.5.8)
   - WCAG Minimum (AA): 24 x 24 CSS pixels
   - WCAG Recommended (AAA): 44 x 44 CSS pixels
   - Platform Standards: 48 x 48 dp (Android), 44 x 44 pt (iOS)
   - Ergonomic Standard: 1cm x 1cm
   - Spacing: Minimum 8px/8dp of dead space between targets
   - High-Impact Sizes for Accessibility: 18mm for motor impairments, 57px for ASD users, 10–15mm for blind users
   - Binary Pass: Targets ≥ 44 x 44 px? Yes/No

4. Keyboard Accessibility (SC 2.1.1)
   - All functionality must be reachable and triggerable via keyboard only
   - Users should be able to navigate the entire site using Tab key
   - Binary Pass: All features work via keyboard? Yes/No

5. Focus Indicators (SC 2.4.7 / 2.4.11 / 2.4.12)
   - Visible focus must be provided for all interactive elements
   - Focus indicators must not be obscured by other content
   - Focus indicators must maintain specific contrast levels (3:1)
   - New SC 2.4.12: Elements like sticky headers must not obscure the focused item
   - Binary Pass: All elements have visible focus indicators? Yes/No

6. Error Identification (SC 3.3.1)
   - Errors must be identified in text
   - Errors must be programmatically associated with the input
   - Use ARIA: aria-invalid="true" paired with aria-describedby pointing to error text
   - Binary Pass: Errors are identified and associated? Yes/No

7. Orientation (SC 1.3.4)
   - Applications must not lock to a single orientation (Portrait or Landscape)
   - Users must be able to rotate devices and content should respond
   - Binary Pass: Both orientations supported? Yes/No

8. Animation/Motion Control (SC 2.3.3)
   - If a background video loops infinitely, a physical pause button must exist
   - Alternatively, videos must stop after 5 seconds maximum (no infinite loop)
   - Binary Pass: Animations can be paused or auto-stop? Yes/No

PART 2: BAYMARD INSTITUTE E-COMMERCE GUIDELINES
================================================================================

Based on 200,000+ hours of large-scale e-commerce testing and 4,400+ test sessions.
Note: Sources reference different numbers of total guidelines: 207, 650, 700+, or 771, 
likely reflecting database updates over time.

1. Guest Checkout Prominence
   - Guest checkout must be the primary, default, most prominent option
   - Forced account creation is a leading cause of cart abandonment
   - Finding: 84% of sites fail to delay account creation until post-purchase
   - Finding: 47% of sites fail to make guest checkout prominent enough
   - Binary Pass: Guest checkout is most prominent option? Yes/No

2. Redundant Data Entry Prevention
   - Never request previously entered data (e.g., email, phone, address) 
     multiple times in a single flow
   - If user entered email on previous page, don't ask again
   - Binary Pass: No redundant data entry occurs? Yes/No

3. Field Grouping / Logical Segmentation
   - Checkout forms should be segmented into logical steps
   - Example: Shipping, Payment, Confirmation
   - Each step should feel like a manageable unit
   - Binary Pass: Fields are logically grouped into steps? Yes/No

4. Inline Validation
   - Errors must be flagged in real-time
   - Validation should trigger "on blur" (when leaving the field), not while typing
   - Use clear, instructional copy for error messages
   - Error messages must disappear the instant a user corrects the input (per-keystroke)
   - Finding: 22% higher task success rate with inline validation
   - Finding: 22% decrease in errors with inline validation
   - Finding: 42% decrease in completion time with inline validation
   - Finding: 31% increase in user satisfaction with inline validation
   - Binary Pass: Inline validation is implemented? Yes/No

5. Product Discovery
   - Product listings must provide at least 3 thumbnails for quick evaluation 
     without leaving the listing page
   - 1–2 images are insufficient for evaluation
   - Visually-driven products require 3 to 15 images
   - Finding: 42% of users judge size from images
   - Binary Pass: 3+ thumbnails visible? Yes/No

6. Search Logic Requirements
   - Autocomplete must be typo-tolerant
   - Autocomplete must support abbreviation mapping (e.g., "3pc" to "3-piece")
   - Autocomplete must return category scopes, not just product names
   - Example: "13 in" should return results for "13 inch"
   - Finding: 77% of mobile sites fail to include category scope in suggestions
   - Finding: 58% of mobile sites fail to support abbreviation/symbol searches
   - Suggestions should appear after 2–3 characters
   - Maximum 5–8 suggestions per dropdown (no internal scrollbars)
   - Binary Pass: Autocomplete is typo-tolerant and category-scoped? Yes/No

7. Price Transparency
   - Total order costs (including shipping and taxes) must be estimated and 
     displayed before the final checkout step
   - Users should never be surprised by fees at the end
   - Binary Pass: Total cost visible before final confirmation? Yes/No

8. Hamburger Menu Avoidance (Desktop)
   - Desktop hamburger menus are an "anti-pattern"
   - Finding: Hamburger menus reduce engagement by ~500% compared to visible links
   - Finding: Hidden menus increase task completion time
   - They should be used only as a "necessary evil" on mobile
   - Binary Pass: Desktop has visible navigation (not hamburger)? Yes/No

A. FIELD-LEVEL OPTIMIZATION

1. Full Name Field (Instead of First/Last Split)
   - Recommendation: Use a single "Full Name" field instead of separate "First Name" 
     and "Last Name" fields
   - Finding: 42% of users type their full name into the "First Name" box if 
     separate fields are used
   - Finding: Only 4% of users hesitate with a single "Full Name" field
   - Finding: 89% of sites still incorrectly use separate First/Last fields
   - Impact: Reduces rework and friction
   - Binary Pass: Single "Full Name" field used? Yes/No

2. Address Line 2 Field
   - Rule: Must be hidden by default
   - Trigger text must be specific: "Add Apt. #, Suite, Floor"
   - Finding: 30% of users pause at Address Line 2 and question whether they 
     need it, even if they don't
   - Finding: 75% of sites fail this guideline (show the field by default)
   - Binary Pass: Address Line 2 hidden behind specific link text? Yes/No

3. Billing Address Field
   - Rule: Default to "Billing = Shipping" and hide redundant billing fields
   - Finding: 24% of sites incorrectly force users to fill out billing address 
     even when it's the same as shipping
   - Binary Pass: Defaults to "Billing = Shipping" with option to change? Yes/No

4. Coupon Code Field
   - Rule: Must be hidden by default behind a text link
   - Link text should read: "Got a promo code?"
   - Finding: Displaying a prominent coupon code field by default causes users to 
     leave the checkout flow to search for codes on external sites
   - Finding: 35% of sites display the coupon field by default (trigger abandonment)
   - Binary Pass: Coupon field hidden behind link "Got a promo code?"? Yes/No

5. Security Code (CVV/CVC)
   - Rule: Must provide a visual hint (thumbnail) showing where the code is on the card
   - Finding: 38% of users pause at CVV fields without a visual hint
   - Finding: Thumbnail dramatically reduces hesitation
   - Binary Pass: Visual hint provided for CVV field? Yes/No

6. Credit Card Auto-Formatting
   - Rule: Input field must auto-format (grouping digits in 4-4-4-4)
   - Finding: 15% of sites fail this guideline
   - Rule: Luhn (Modulus 10) checksum validation should be performed in real-time
   - Users should be able to verify an unbroken 16-digit string
   - Binary Pass: Auto-formatting applied to card numbers? Yes/No

7. Field Count and Layout
   - Average checkout: 11.3 to 12.8 fields
   - Optimized checkout: 6 to 8 fields
   - Reduction potential: Most sites can reduce visible fields by 20–60%
   - Case Study (Expedia): Removing one unnecessary "Company" field increased 
     annual profits by $12 million
   - Finding: Single-column layouts completed 15.4 seconds faster than multi-column
   - Finding: Multi-column layouts cause users to skip fields or break visual momentum
   - Finding: 16% of sites use multi-column forms (suboptimal)
   - Binary Pass: ≤8 visible fields in guest checkout? Yes/No

8. Input Label Alignment
   - Rule: Top-aligned labels are processed 28% faster than left-aligned labels
   - Binary Pass: Labels are top-aligned? Yes/No

1. Cart Feedback and Confirmation
   - Rule: Use numeric badges on cart icon to show item count and subtotal
   - Rule: When item is added, use a persistent overlay (not transient/fading)
   - Finding: Transient overlays cause a "race against time" feeling and user anxiety
   - Cart subtotals and item counts should be visible on the cart icon badge
   - Binary Pass: Cart badge shows item count/subtotal? Yes/No

   - Rule: Provide +/- buttons AND direct keyboard entry
   - Rule: Never use dropdowns for quantity selection alone
   - Finding: 61% of sites fail this guideline (use dropdowns only)
   - Impact: Dropdowns are inefficient for quantity selection
   - Binary Pass: +/- buttons with text input provided? Yes/No

   - Rule: Remove explicit "Apply" or "Update" buttons for form changes
   - Changes should persist automatically on blur or change event
   - Finding: 22% of sites violate this (include unnecessary "Apply" buttons)
   - Binary Pass: Changes auto-persist without needing apply button? Yes/No

4. Reset/Clear Buttons
   - Rule: NEVER use a "Reset" or "Clear Form" button
   - Binary Rule: If a "Reset" button exists, it is a catastrophic failure
   - Danger: Accidental clicks cause complete data loss, leading to abandonment
   - Rule: If a "Reset" button must exist, it must be physically separated from 
     "Submit" by at least 200px, or removed entirely
   - Binary Pass: No "Reset" button exists or is >200px from "Submit"? Yes/No

5. Account Creation Timing
   - Rule: Always delay account creation until AFTER the purchase is complete
   - Finding: 84% of sites fail this guideline (force account creation at checkout)
   - Finding: Forced registration is a primary abandonment cause
   - Impact: Post-purchase account creation gives users a reason to complete the transaction
   - Binary Pass: Account creation delayed until post-purchase? Yes/No

6. Guest Checkout Prominence
   - Rule: Guest checkout must be the most prominent, default option
   - Rule: Never use confusing language like "Continue as Guest" after showing

   - Binary Pass: Guest checkout is the primary, prominent option? Yes/No

7. Progress Indicators in Multi-Step Flows
   - Rule: Linear, non-branching progress indicators reduce user anxiety
   - Finding: Progress bars correlate with reduced abandonment
   - Finding: Some reports suggest 10–15% conversion lift from implementing clear 
     progress bars
   - Rule: Progress bar must show maximum 3–4 steps
   - Rule: If process requires 5+ steps, redesign into a multi-step wizard, not 
     just a progress bar
   - Concern: Poorly designed indicators (too many segments) can actually increase 
     abandonment by showing how far users have to go
   - Binary Pass: Progress indicator shows ≤4 steps? Yes/No

8. Conversion Lift Potential
   - Finding: Improving checkout UX can lead to a 35.2% increase in conversion rates
   - Finding: This 35.2% is an aggregate of 32 different heuristics combined

PART 10: DOCUMENTED CONTRADICTIONS, CONFLICTS, AND UNRESOLVED GAPS
================================================================================

1. THE "35% CONVERSION LIFT" CONTRADICTION
   - Claim: 35.2% conversion lift from checkout optimization
   - Reality: This is an AGGREGATE of 32 different heuristics
   - Clarification: NO single heuristic (e.g., "blue button vs. red button") 
     provides this level of gain
   - Important Distinction: The aggregate of all optimizations combined yields the lift
   - Missing Data: No public, peer-reviewed study isolates single form field changes 
     and proves causal conversion lift via controlled A/B test data

2. FITTS'S LAW ON TOUCH DEVICES
   - Academic Debate: Some argue Fitts's Law doesn't apply identically to touchscreens
   - Reason: Finger occlusion (the finger hides the target)
   - Status: Unresolved; different interpretations exist

3. VALIDATION ERROR PRESENTATION CONFLICT
   - Browser-native behavior: Stops at first error, prevents further submission
   - UX best practice: Show all errors simultaneously
   - Conflict: These approaches contradict each other
   - Status: No definitive resolution; both approaches have merit

4. PROGRESS INDICATOR EFFICACY CONTRADICTION
   - General finding: Progress indicators reduce abandonment
   - Contrary finding: Poorly designed indicators (too many steps) can INCREASE 
     abandonment
   - Mechanism: Too many steps shows users how far they have to go, triggering fatigue
   - Resolution: Limit progress bars to ≤4 steps; use multi-step wizards for 5+ steps

5. SERP FEATURES (SEARCH ENGINE RESULTS PAGE) CONTRADICTION
   - Rich snippets: Increase CTR (click-through rate)
   - Knowledge Graphs: Can LOWER CTR for the top organic result
   - Reason: Knowledge graphs answer the question on the SERP, removing the 
     incentive to click through
   - Status: Both findings validated in different contexts

6. BAYMARD GUIDELINE COUNT DISCREPANCY
   - Cited numbers: 207, 650, 700+, or 771 guidelines
   - Explanation: Baymard's database is a living, tiered product
   - Status: Different numbers reflect database updates over time
   - "Master list" is not static; counts change as research evolves

7. TOTAL RESEARCH HOURS DISCREPANCY
   - Cited numbers: 200,000 hours, 150,000 hours, or 4,400+ test sessions
   - Status: Numbers conflict; likely reflect different research phases
   - Unclear: What counts toward the total hours claim
   - Note: Hours and session counts may measure different things

8. SINGLE "FULL NAME" FIELD CONFLICT
   - UX Best Practice: Single "Full Name" field is best (42% vs. 4% hesitation)
   - Marketing Tool Conflict: Facebook Pixel and similar tools prefer split First/Last
   - Issue: Data structure incompatibility between UX optimization and marketing tools
   - Status: Unresolved; requires compromise or custom data transformation

9. "RESET BUTTON" RULE CONTRADICTION
   - Finding: Reset buttons cause catastrophic accidental data loss
   - Binary Rule: Reset buttons must NEVER be used
   - Counter-need: Some users want to clear a form and start over
   - Resolution: Use "Start Over" or "Clear All" link (not a button), preferably 
     with confirmation

10. MOBILE TARGET SIZE CONTRADICTION
    - WCAG Standard: 44 x 44 CSS pixels (AA)
    - Android Material Design: 48 x 48 dp
    - iOS HIG: 44 x 44 pt
    - Ergonomic Standard: 1cm x 1cm (approximately 35–40 pixels)
    - All cited sources cite different values
    - Binary Rule: The LARGEST minimum of these is 48 x 48, which is the safest 
      for cross-platform compliance

11. HEURISTIC DEFINITION CONFLICT
    - Nielsen/Molich: 10 usability heuristics (high-level principles)
    - Baymard: 207–771 heuristics (component-level requirements)
    - These are NOT the same category
    - Status: Terminology conflict (both use "heuristics" for different things)

12. MISSING PEER-REVIEWED DATA
    - Most "CRO rules" found in blogs: Professional opinions or industry standards

    - Exception: Some findings (load time/bounce, inline validation) have
